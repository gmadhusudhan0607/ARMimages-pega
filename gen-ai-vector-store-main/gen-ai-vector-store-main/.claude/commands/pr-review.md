**Quick mode** — single-context review, dobre dla małych zmian. Dla pełnego feature review użyj agentów `reviewer` + `security-reviewer` równolegle (głębsze, VS-specific checks).

Perform a thorough code review of the current branch's changes compared to `main`.

## Steps

1. Get the full diff and changed files:
   ```bash
   git diff main...HEAD --stat
   git diff main...HEAD
   ```

2. Read every changed file in full to understand context (not just the diff hunks).

3. Review all changes against the checklist below.

4. Output findings in the format specified at the end.

## Review Checklist

### Architecture & Structure
- Changes follow established directory structure (`cmd/service/`, `cmd/ops/`, `cmd/background/` for entry points, `cmd/middleware/` for shared middleware, `internal/` for core logic)
- API versioning respected - breaking changes require new version (`v1` -> `v2`)
- OpenAPI specs in `apidocs/` updated if endpoints changed
- No new `main.go` files outside `cmd/` subdirectories

### Go Code Quality
- Copyright header on all new Go files: `/* Copyright (c) <year> Pegasystems Inc. ... */` (block comment style)
- `snake_case.go` file naming convention
- No bare `panic` - handle errors explicitly
- No mutable package-level state without synchronization
- Proper error wrapping: `fmt.Errorf("context: %w", err)`
- `context.Context` as first parameter where needed (DB calls, HTTP calls)
- Proper `defer` for resource cleanup (DB connections, HTTP response bodies)
- No dead code, debug code, or TODO comments in production code

### Logging & Observability
- Uses `log.GetNamedLogger("name")` for zap logger initialization
- Structured fields: `logger.Info("msg", zap.String("key", val))`
- No `fmt.Print*`, `log.Print*`, or `println` in production code
- Exceptions logged with stack context
- No sensitive data (passwords, tokens, PII) in logs

### Error Handling
- Fail fast - return errors to client, no silent fallbacks
- Proper error wrapping with context at each level
- Correct HTTP status codes (200, 201, 400, 401, 404, 500)
- Consistent error response format

### Database (pgx/pgxpool)
- Uses `pgxpool.Pool` for connections (never standalone `pgx.Conn` for production)
- Bulk queries preferred over N+1 patterns
- SQL functions used where they reduce complexity
- Proper transaction handling with rollback on error
- Connection/context passed through, not created ad-hoc
- No SQL injection risks (parameterized queries only)

### Configuration
- Uses `helpers.GetEnvOrDefault()` for env var access
- Follows `internal/config` patterns for structured config
- No hardcoded secrets - loaded from environment
- `docs/environment-variables.md` updated if env vars changed

### Testing
- Unit tests present alongside source code (`*_test.go`)
- Standard `testing` package for unit tests
- Ginkgo/Gomega for integration tests in `src/integTest/`
- Mocks generated by `go tool mockery` (no manual mocks)
- Happy path, error cases, and edge cases covered
- External dependencies mocked
- `ExpectNoIdleTransactionsLeft` in `AfterSuite` for DB tests

### Dependencies
- Only approved libraries (gin, zap, pgx/v5, ginkgo/gomega, prometheus, aws-sdk-go-v2, go-sax)
- No new HTTP frameworks, logging libraries, or database drivers
- `go.mod` changes minimal and justified

### Security
- Input validation on all request parameters
- No command injection, path traversal, or SSRF vulnerabilities
- SAX authentication enforced where required
- Parameterized SQL queries (no string concatenation)

### Performance
- Minimize DB round-trips (bulk operations, SQL functions)
- No unnecessary goroutine creation
- Proper connection pool usage (no pool exhaustion patterns)
- Context timeouts for external calls

### Backwards Compatibility
- Existing tests still pass
- Existing endpoints still work
- No unrelated changes included

## Output Format

### Critical Issues (must fix before merge)
- `file:line` - description, why it matters, suggested fix

### Major Issues (should fix)
- `file:line` - description, impact if not fixed, recommendation

### Minor Issues / Suggestions
- `file:line` - suggestion, why it would be better

### Security Concerns
- `file:line` - issue, potential impact, how to fix

### Positive Observations
- What was done well and why it is valuable

### Educational Notes
- Patterns or concepts worth learning from this review
